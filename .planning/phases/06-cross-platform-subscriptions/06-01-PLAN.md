# Plan 06: Cross-Platform Subscription Overhaul

## 1. Objective
Migrate the application from an insecure, policy-violating web-redirect subscription model to a secure, compliant, and platform-native monetization system.

## 2. Technical Strategy
- **Android, iOS, macOS:** Use the `in_app_purchase` Flutter package to integrate with Google Play Billing and Apple App Store In-App Purchases.
- **Windows:** Integrate Razorpay via a secure direct API or a localized web-view flow, as Windows does not have a native IAP store system for most Flutter desktop apps.
- **Backend:** Centralize Pro status logic in a strictly server-controlled `subscriptions` table, removing all client-side metadata triggers.

---

## 3. Phase Breakdown

### Phase 1: Setup & Branching (Wave 1)
- **Branching:** Create a new branch `feat/cross-platform-subscriptions`.
- **Dependencies:** Add `in_app_purchase` to `pubspec.yaml`.
- **Infrastructure:** Initialize `IAPService` to handle the platform-specific purchase streams.

### Phase 2: Backend Hardening (Wave 1)
- **Surgical SQL Cleanup:**
    - **Delete** the `handle_user_metadata_update` function and its trigger on `auth.users`.
    - Consolidate `MASTER_DATABASE_SCHEMA_FINAL.sql` and all `FIX_*.sql` files into a clean `supabase/migrations/` structure.
- **Secure Schema:**
    - Ensure the `subscriptions` table is properly defined with RLS.
    - Implement a `validate_purchase` RPC or Edge Function that verifies receipts/payments before updating `profiles.is_pro`.
- **Edge Function Updates:**
    - Update `paypal-verify` and `razorpay-verify` to validate that `amount_paid == plan_price`.

### Phase 3: Mobile & macOS Native IAP (Wave 2)
- **Implementation:**
    - Implement `InAppPurchase` initialization and product loading.
    - Handle `PurchaseDetails` stream and "Complete" transactions.
    - Implement **Server-Side Receipt Validation** (connecting to Google/Apple APIs via Supabase Edge Functions).
- **Compliance:** Ensure the "Restore Purchases" functionality is present for iOS/macOS.

### Phase 4: Windows Razorpay Integration (Wave 2)
- **Implementation:**
    - Create a Windows-specific Razorpay flow using a secure `webview_flutter` or similar container.
    - Connect directly to the existing `razorpay-create-order` and `razorpay-verify` Edge Functions.

### Phase 5: Unified Security Audit (Wave 3)
- **Global Refactor:** Search and replace all instances of `userMetadata['is_pro']` with calls to `SubscriptionService().isPro`.
- **Validation:** 
    - Ensure `WellnessService`, `VaultService`, `CanvasService`, and `StoryService` all respect the new secure Pro status.
    - Verify that forging metadata no longer grants access.

---

## 4. Verification Plan
- **Security:** Attempt to bypass the paywall via `supabase.auth.updateUser` (should fail).
- **Compliance:** Verify that Android/iOS no longer redirect to external URLs for payment.
- **Flow:** Test end-to-end subscription flow on:
    - Android Emulator (Play Billing)
    - iOS Simulator (Sandbox IAP)
    - macOS (Sandbox IAP)
    - Windows (Razorpay API)
